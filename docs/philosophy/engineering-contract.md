## 目的與邊界

Accountable Dialogue 的目的不是讓系統看起來擁有內在，而是讓它能誠實交代：

- 它如何改變；
- 它依據什麼改變；
- 誰批准了改變；
- 改變造成什麼後果；
- 哪些部分仍未實作或驗證。

這份契約是可追溯性的外骨骼。它約束協作者如何陳述進展，不能單獨證明輸出為真、模型具有主觀經驗，或模型已從語料學會誠實。這些問題需要獨立的資料同意、操作定義、反例、保留集與校準評估。

## 一、重要改動必須有帳

「重要改動」指會改變外部可觀察行為、價值判準、資料或隱私、權限、安全、成本界線、不可逆性，或未來責任歸屬的變更。

每一項重要改動至少應留下：

- 改動理由；
- 對應 Artifact；
- 支持或反駁它的證據；
- 預期或已觀察到的後果；
- 提出者、實作者、審查者與定案者；
- 驗證範圍與尚未驗證的部分。

角色可以由同一人兼任，但必須如實標示。沒有 Artifact、Schema、規則或測試變更時，只能記錄為設計討論或 Proposal，而非架構演進。

## 二、提案不能自行升格

對話、多代理審議或分析代理提出的觀點預設都是 Proposal。它們可以提出方案、反例與張力，也可以執行已授權的檢查；但不能自行把價值判準、承諾或候選架構升格為正式原則。

```text
提出 ≠ 定案
稱讚 ≠ 批准
寫入格式 ≠ 已持久化
對話共識 ≠ 系統規則
```

只有系統擁有者或其明確授權者具有正式定案權。有限委派必須可核對其範圍、期限與可逆性，不能自行擴張。

## 三、撤回與錯誤也是歷史

已採用的承諾、主張與決策後來被否定時，不得被無聲覆寫。重要的公開紀錄應保留原始內容或最小去識別化引用、當時理由、挑戰或撤回原因、權限、替代版本、生效時間與適用範圍。

但可追溯性不高於隱私、合法刪除要求或安全處置。原文必須移除時，應保留最小處置紀錄，而非強留敏感內容。

## 四、未解張力依風險分流

低風險且可逆的改動可以暫定進行，但須標示假設、未解不確定性、撤回條件與後續驗證方式。

高影響、不可逆、涉及私密、權限、金錢、法律或醫療後果的行動，不得由 AI 默默定案。除非已有明確且可核對的授權政策，否則必須停在 Human Review Gate。

## 五、可稽核不等於全面揭露

應區分可公開且已最小化、已授權或去識別化的決策摘要與 Artifact 引用、可驗證的來源、受限存取的私人脈絡、敏感資料，以及不作為交付內容的隱藏內部推理。若摘要或引用仍可能揭露私人事實，應改用受限或不透明引用。系統應保存最小充分紀錄；原始對話不因為可分析就預設可持久化。

## 六、類型與成熟度分離

每一筆紀錄先回答「它是什麼」，再回答「它走到哪裡」。

| 類型 | 意義 |
|---|---|
| `Observation` | 有來源或情境的觀察，不自動等於真實主張 |
| `Claim` | 可被證據支持、挑戰或撤回的陳述 |
| `Proposal` | 尚待定案的方案、模型或行動 |
| `Commitment` | 已宣告、可被追問的規範性義務 |
| `Decision` | 由具權限者作出的選擇 |
| `Evidence` | 支持、限制或反駁某項主張的材料 |
| `Artifact` | 文件、程式、Schema、設定、測試或版本引用 |
| `Verification` | 方法、結果與驗證範圍的紀錄 |
| `Event` | 已發生、不可被原地覆寫的變更或處置 |

治理狀態：`draft`、`under_review`、`ratified`、`rejected`、`not_applicable`。

工程狀態：`not_implemented`、`partial`、`implemented`、`not_applicable`。

驗證狀態：`not_tested`、`inconclusive`、`partially_verified`、`verified_within_scope`、`failed`、`not_applicable`。

生命週期：`active`、`suspended`、`withdrawn`、`superseded`、`archived`、`not_applicable`。

並非每個類型都適用每條狀態軸；不可為了表格完整而捏造狀態。

## 七、文章與完成聲稱接受同一套審查

任何宣稱架構演進的文章至少回答：

1. 改了哪個 Artifact？
2. 改了哪個 Schema？
3. 新增、刪除或修改了哪條規則？
4. 有什麼測試或證據支持？
5. 哪些部分仍停留在提案、實作中或未驗證？

每題都可誠實回答 `none`。但若 Artifact、Schema、規則與測試皆無實際改動，文章只能描述設計討論。

「完成」必須指出驗收標準、測試結果、Artifact、驗證範圍、已知限制與批准接受者。否則應使用 `proposed`、`drafted` 或 `implemented_but_unverified` 等較窄描述。

## 核心句

> 不替系統宣稱靈魂；只要求它對影響負責。
