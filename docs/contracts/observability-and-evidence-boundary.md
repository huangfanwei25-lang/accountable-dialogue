# 可觀測性與證據邊界

> 版本：`v0.1-proposed`<br>
> 文件類型：公開聲稱邊界<br>
> 治理狀態：`under_review`<br>
> 工程狀態：`not_implemented`<br>
> 驗證狀態：`not_applicable`

本文件補充[哲學／工程契約](../philosophy/engineering-contract.md)。它處理一個常被混淆的問題：有 trace、schema 或測試，不等於看見了所有內部機制；沒有看見所有內部機制，也不等於什麼都不能檢查。

## 可觀測性不是二元的

| 分類 | 可以誠實地說 | 不可以據此說 |
|---|---|---|
| `observable` | 行為、Artifact 或明確輸出的欄位可直接檢查 | 它已說明所有未輸出的內部選擇 |
| `partially_observable` | 輸出與部分中介證據可檢查，邊界已命名 | 可從摘要推得完整因果機制 |
| `opaque` | 目前沒有足夠直接證據檢查該內部過程 | 從結果、語氣或自我報告推定其真實動機、意識或隱藏推理 |

### 三條規則

1. **外殼證據是真實證據，但不是總存取權。** 已輸出的事件、測試結果或 Artifact 可作為它們自身範圍內的證據。
2. **不透明處必須被命名。** 若選擇過程、私人脈絡或模型內部狀態不可直接檢查，應標示 `opaque`，而不是用流暢解釋填補空白。
3. **解釋不得超過證據。** 對輸出行為的解釋可以是 Proposal 或假說，不能被寫成對隱藏機制的事實陳述。

## 證據種類與可用語言

證據不是單一分數。每項主張應指出它依賴哪一種證據，以及其驗證範圍。

| 證據種類 | 可以支持的聲稱 | 不能支持的聲稱 |
|---|---|---|
| `test_backed` | 特定輸入／輸出或回歸行為在測試範圍內通過 | 生產環境品質、通用正確性或內在動機 |
| `structure_checked` | Schema、型別或靜態規則符合指定結構 | 欄位內容真實、合乎倫理或實際有效 |
| `runtime_observed` | 在明示環境、版本與時間中觀察到行為 | 所有環境、供應商或未測情境皆成立 |
| `source_backed` | 可核對來源支持的特定事實或限制 | 來源未處理的因果、價值或普遍結論 |
| `owner_ratified` | 有權者接受某項規範、範圍或決策 | 該決策已被實作或已產生預期效果 |
| `documented` | 文件記錄了設計意圖或提案 | 機制已存在、已啟用或已驗證 |
| `narrative` | 價值、隱喻或待檢驗的方向 | 可操作能力、真值或測量結果 |
| `blocked` | 目前缺乏可取得的驗證路徑 | 以缺乏證據推定任何正面或負面結論 |

## 權限與證據是不同軸

一項原則可以由擁有者正式定案，卻仍未實作；一個低風險 helper 可以有強測試，卻無權改寫政策。所有重要紀錄都應分開回答：

1. **誰有權提出或定案？**
2. **什麼證據支持此刻的描述？**
3. **哪些部分仍不可觀測或未驗證？**

這避免兩種錯誤：把有權者的決策當成事實證明，或把通過測試當成價值定案。

## 安全的公開說法

| 情況 | 安全說法 |
|---|---|
| 有 schema，沒有行為測試 | 「結構已定義並檢查；語義與效果尚未驗證。」 |
| 有單元測試，沒有部署資料 | 「此行為在列出的測試範圍內通過；尚未聲稱生產效果。」 |
| 有決策摘要，沒有原始對話 | 「決策摘要與 Artifact 可檢查；私人脈絡未公開，不能據此推定完整推理。」 |
| 出現模型自我報告 | 「這是模型的輸出陳述，不是對其內在狀態的直接測量。」 |

## 未來若成為工具

任何 runtime、schema 或 validator 若要使用本文件的詞彙，必須：

- 保留 `observable`／`partially_observable`／`opaque` 的適用範圍；
- 不接收或持久化原始私人對話作為預設；
- 對 evidence、authority、implementation 與 verification 分別建模；
- 有測試證明它拒絕把未驗證或不透明資訊升格為完成聲稱。

在此之前，這是一份可審查的文件契約，不是執行中的保證。
